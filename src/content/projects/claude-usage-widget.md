## Summary

A native macOS desktop widget built in one day (~5 hours) entirely with Claude Code. It displays real-time Claude AI usage limits — current session percentage, weekly all-models usage, and weekly Sonnet usage — directly on the desktop. The widget auto-refreshes every 60 seconds, keeps the session alive via background WKWebView refresh, and launches at login as a pure background agent (no Dock icon, no menu bar icon).

## Features

- Native macOS WidgetKit widget (small and medium sizes)
- Google Workspace authentication via embedded WKWebView
- Auto-refresh every 60 seconds with smooth animated progress bar
- Background session keep-alive every 4 hours (no re-login needed)
- Official Claude starburst logo rendered from SVG
- Pure background agent — no Dock icon, no menu bar
- Launch at Login via SMAppService
- Click widget to open control panel (Refresh / Sign Out / Quit)
- Dark theme with high contrast (WCAG AA compliant)
- Opus usage column appears automatically when available

## How It Was Built

This project was built from scratch in a single day across two sessions, entirely through conversation with Claude Code. The process started with an interview phase where Claude asked about the tech stack, auth approach, and design preferences before writing any code. From there, it was iterative: Claude would implement a feature, I'd test it and share screenshots, and we'd refine.

The biggest challenge wasn't building the widget itself — it was getting two sandboxed processes (the app and widget extension) to share data on macOS. This led to a deep debugging session that uncovered fundamental limitations with App Groups on free Apple Developer accounts, ultimately solved with a creative file-based approach.

The entire codebase (1,376 lines of Swift across 15 files) was generated by Claude Code, with me providing direction, screenshots, and testing feedback.

## Technical Challenges

### The App Groups Saga (Widget Shows "Open app to sign in")

**Problem:** After successful login, the widget continued showing "Open app to sign in." The app was saving auth data, but the widget extension couldn't read it.

**Root cause discovery (multi-step):**

1. **First hypothesis — Cloudflare blocking:** The widget's URLSession API calls to claude.ai were returning 403. The error handler was clearing auth on 403, wiping credentials immediately after saving them. Fixed by switching all API calls to WKWebView-based JS fetch (same-origin bypasses Cloudflare).

2. **Second hypothesis — App Sandbox mismatch:** The main app wasn't sandboxed but the widget extension was. They were writing/reading UserDefaults from different locations. Added sandbox to the main app.

3. **Third hypothesis — App Group not provisioned:** Even with both targets sandboxed and the entitlement in the binary, the provisioning profiles didn't include `application-groups`. The App Group was never registered on the Apple Developer Portal. `xcodebuild -allowProvisioningUpdates` couldn't auto-register it.

4. **Discovery — Free developer account limitation:** "App Groups" didn't appear in Xcode's capability list. Free Apple Developer accounts don't support App Groups.

5. **Final solution — Direct file-based sharing:** The main app (non-sandboxed) writes a JSON file directly into the widget extension's sandbox container at `~/Library/Containers/com.fdelgado.claudeusage.widget/Data/Library/Application Support/shared_state.json`. The widget extension (sandboxed) reads from its own Application Support directory — same physical file. No App Groups needed.

**Debugging approach:** Checked `~/Library/Group Containers/`, `~/Library/Preferences/`, `~/Library/Containers/`, examined provisioning profiles with `security cms -D`, verified codesign entitlements, streamed system logs with `log show`/`log stream`, and tested file write access to the widget container.

### Cloudflare Blocking URLSession

**Problem:** All URLSession requests to `claude.ai/api/...` returned 403 (Cloudflare protection), even with proper cookies and Safari user-agent.

**Solution:** All API calls go through a WKWebView's JavaScript context using `callAsyncJavaScript`. Since the WKWebView has a valid session and the fetch is same-origin, Cloudflare doesn't block it. This applies to the login flow, periodic refresh (via SessionRefresher's hidden WKWebView), and the initial usage fetch after login.

### Keychain Password Prompts

**Problem:** The widget extension accessing macOS Keychain triggered authorization dialogs 3 times in a row, even after clicking "Always Allow."

**Solution:** Removed Keychain entirely. Moved all auth storage to the file-based SharedDataService. Widget extensions are separate processes with separate Keychain access, which is why the prompts appeared — each access from the extension required user authorization.

### Widget Extension Caching

**Problem:** After rebuilding and reinstalling the app, the widget continued showing old UI (wrong icon, old animation). Code changes appeared to have no effect.

**Solution:** The widget extension is a system-managed process that persists independently of the main app. Quitting the app doesn't kill the extension. Must explicitly `pkill -9 -f "UsageWidgetExtension"` to force the system to relaunch it with the new binary.

### Smooth Progress Animation Without Gray Track

**Problem:** WidgetKit doesn't support custom animations. Custom `ProgressViewStyle` broke the native smooth interpolation, causing 1fps stepping. But the default `.linear` style shows a gray track background.

**Solution:** Use the default system `ProgressView(timerInterval:)` with `.linear` style (preserves native smooth animation), then apply `.brightness(-0.15)` and `.contrast(5)`. This pushes the dim gray track to black (invisible on the black widget background) while keeping the bright blue fill visible.

## Architecture

Key technical decisions:

- **WKWebView for all API calls** — URLSession is blocked by Cloudflare. All fetches use `callAsyncJavaScript` within a WKWebView context (same-origin).
- **File-based data sharing** — App Groups require a paid developer account. Instead, the non-sandboxed main app writes directly to the widget extension's sandbox container.
- **Persistent WKWebView cookies** — Using `.default()` data store (not ephemeral) so session cookies survive app restarts. SessionRefresher loads claude.ai every 4 hours in a hidden WKWebView to keep the session alive.
- **Single timeline entry + system ProgressView** — Instead of 60 timeline entries per minute, uses one entry with `ProgressView(timerInterval:)` for native smooth animation.
- **SVG logo via NSImage** — The Claude starburst logo SVG is embedded as a string literal, rendered via `NSImage(data:)` at runtime. Avoids asset catalog issues across targets.
- **Background agent** — `LSUIElement = true` (no Dock icon) + `SMAppService.mainApp.register()` (Launch at Login) + no visible scenes.
